⚡ 短期方案 两阶段搜推管道：粗召回 → 五维精排 → 多因子重排序

阶段一：粗召回（Candidate Generation）

目标是以低成本筛出"值得精排"的候选集，宁可多召回也不能漏。当前规则：

过滤条件 当前参数 设计理由
时效窗口 最近 30 天 情报的时效性窗口；超过30天的内容对战略决策参考价值有限
评分阈值 criteria_score ≥ 55 排除明确无关内容，55分以上进入聚类候选；低于此值的内容仍存库，可查询
候选上限 最多 150 篇 LLM聚类上下文限制；150篇分4批处理，每批≤40篇
来源层级优先 关键玩家 顶尖研究者 优先入候选集 这两类来源无论评分高低，信号质量更可信；避免被媒体聚合文章稀释
⚠️
当前瓶颈：粗召回是纯规则过滤，没有语义理解。一篇高度相关但措辞特殊的文章可能因LLM误判（得分偏低）而被漏掉。这是长期方案中向量召回需要解决的问题。

阶段二：五维评分精排

召回后的每篇文章，由 DeepSeek 模型对照京东业务评分规则打5个维度的分，输出一个总分和打分理由。五个维度的设计逻辑：

维度 满分 衡量内容 京东视角的重点
相关性
relevance		 30 与京东14个商业领域的直接关联程度 重点加权：智能零售/物流/搜推/广告/金融；对纯通用AI内容相关性打分偏低
新颖性
novelty		 25 信息增量 — 是新知识、新趋势，还是已知内容的重复 RT（转推）硬封顶12分；社区讨论封顶15-20分（有一手测试数据的上浮）
收敛度
convergence		 20 该信号是否已被多个独立来源验证 单一来源最高15分；有2个以上独立来源交叉印证，可获满分
来源层级
source_tier		 15 信源本身的权威性和可信度 Tier 1（官方博客/论文/政府公告）满分；Tier 3（聚合媒体/转载）≤8分
时效性
timeliness		 10 发布时间 + 内容是否描述当前正在发生的变化 48h内发布满分；7天内8分；超过14天且无新增信息4分以下
# 总分计算（由LLM输出，非加权求和；LLM有整体判断空间±5分） total_score = relevance + novelty + convergence + source_tier + timeliness # 满分 100 分；LLM fallback（解析失败时）默认 30 分 # 特殊规则覆盖（精排后的score二次修正） if is_retweet: novelty = min(novelty, 12) # RT硬封顶 if feed == 'jd-manual-report': relevance *= 1.3 # 内部报告相关性溢价 if feed == 'jd-manual-community': novelty_cap = 20 if has_evidence_markers else 15 # 社区信号封顶 # evidence_markers = ['实测','内测','benchmark','我们用','我发现']
💡
评分规则文件：LLM的打分依据来自 jd_scoring_rules.md，完整定义了14个业务领域的相关性权重和各维度的评分细则。评分规则可以独立修改而无需改代码。

阶段三：多因子重排序

LLM给出基础分后，系统在Python层叠加三个额外信号进行重排序：

1
X（Twitter）背书乘数

若文章被X关键人物转发/分享，文章的基础分乘以背书权重之和。例如：Karpathy（×0.30）+ Sam Altman（×0.25）共同转发同一篇文章，基础分60分 → 60 × (1 + 0.55) = 93分。上限2.0×（即最高翻倍）。权重由 X_ENDORSER_WEIGHTS 字典维护，按认知权威性分级（0.10–0.30）。

2
收敛聚类加权

把高分文章（≥55分）送入DeepSeek做收敛聚类：找出2篇以上、来自不同立场视角、独立指向同一底层趋势的文章组合。聚类内文章的得分在展示层会被聚类收敛分加权（聚类越强、立场越多样，权重越高）。聚类算法依赖三个乘数：立场多样性、高价值立场出现次数、Tier1文章存在与否。

3
立场多样性加权

相同话题下，若同时有"资本动向 + 顶尖研究者 + 关键玩家"三类高价值立场，其收敛分的乘数系数累加（每多一个高价值立场 +8%，每多一个普通立场 +12%）。纯科技媒体报道群不算收敛，不加分。

# 聚类收敛分公式 base = avg(article_scores) sp_mult = 1.0 + 0.12 × (unique_standpoints - 1) + 0.08 × hv_standpoints tier1_mult = 1.2 if any_tier1 else 1.0 recency_mult= 1.3 (≤7d) | 1.15 (≤14d) | 1.0 (older) size_mult = 1.0 + 0.05 × max(0, cluster_size - 2) x_mult = min(1.30, 1.0 + 0.08 × endorsed_articles + 0.04 × total_endorsers) convergence_score = min(100, base × sp_mult × tier1_mult × recency_mult × size_mult × x_mult)

当前方案的局限性

问题具体表现长期方案如何解决
权重人工设定 5个维度的满分分配（30/25/20/15/10）是经验判断，未经数据验证 Learning to Rank模型从专家反馈数据中学习最优权重
通用性框架 LLM不知道"京东搜推团队今年在押注什么"，无法针对内部战略优先级评分 专家反馈标注注入业务语境，训练数据本身携带京东视角
召回是硬阈值 一篇措辞冷僻但高度相关的文章可能被LLM低估（得分<55）而进不了聚类 Dense Retrieval向量召回补充语义相似度，不依赖LLM评分
X背书匹配率低 当前只有约3篇文章被匹配到DB中的推文链接（外链文章未入库） 扩大外链文章抓取范围；建立URL-to-article索引
🔭 长期方案 参考业界标准：两路召回 → LTR精排 → 差异化多目标

短期方案的核心问题是"权重是猜出来的"。长期方案的根本改变是：让系统从专家反馈数据中学习权重，使排序模型真正理解"什么样的文章对京东CTO团队有战略价值"。

粗召回升级：稀疏检索 + 稠密向量检索双路并行

当前的规则过滤（阈值+时效）只看LLM打的分，无法发现"分数低但语义高度相关"的文章。两路并行可以覆盖不同类型的相关性：

召回策略 工作原理 擅长捕捉 局限
BM25 稀疏检索 基于TF-IDF关键词加权匹配，针对JD战略词库（搜推/物流/具身智能/BNPL…）的命中率打分 精确术语、产品名称、特定公司名、政策法规条文 同义词、跨语言表达（英文提到"recommendation system"但中文索引）
Dense Retrieval 向量召回 用领域embedding模型（如BGE-M3）将文章和查询向量化，通过ANN近邻搜索召回语义相近候选 隐式相关、跨语言语义、新概念（如"vibe coding"与"开发效率"的关联） 需要维护向量索引；embedding模型需要定期更新以捕捉新概念

两路召回的结果通过 倒数排名融合（Reciprocal Rank Fusion, RRF） 合并，生成统一候选集送入精排阶段。RRF是业界通行的无需参数调整的融合方法。

📐
实现路径：向量索引可使用 FAISS（本地）或 Qdrant（服务化），embedding 模型推荐 BGE-M3（支持中英文混合），与现有 SQLite 数据库并列存储。迁移到向量召回无需重写现有评分逻辑，属于并行追加。

精排升级：Learning to Rank（LambdaRank）

LTR是搜索引擎和推荐系统中的标准精排框架。核心思想：不预设权重，而是从带标注的"哪篇比哪篇更重要"数据中，学习出一个最优排序函数。

1
特征工程 — 每篇文章的特征向量

文章的各维度评分（5个维度）+ 来源元数据（tier、standpoint）+ 话题embedding相似度 + X背书权重 + 时效衰减系数 + 历史上相似文章被专家标注为"有价值"的比例。

2
训练数据 — 专家反馈标注（来自第三步）

专家在 JoySpace 文档或长期RSS平台上的行为：标注"有价值/无关/领域错误"，以及转发给产研团队的行为，构成 pairwise 标注对："文章A比文章B对我们更重要"。每积累约200条有效标注即可训练初版模型。

3
模型选型 — LightGBM with LambdaRank objective

LambdaRank是目前业界最主流的LTR算法之一（微软、谷歌均有使用），基于LightGBM实现可在CPU上秒级训练，无需GPU。训练完成后推理速度极快，可替换现有手工multiplier层。

4
迭代节奏 — 月度模型更新

每月汇总新增专家标注，重新训练模型。通过A/B测试或离线指标（NDCG@10）评估新模型效果，确认提升后替换线上版本。早期（标注量<200条）维持现有规则体系，模型作为shadow scorer并行跑验证。

# LambdaRank 目标函数（概念） # 对每对文章(i, j)，若专家标注i比j更重要，则最大化以下： λ_ij = -∂NDCG/∂s_i × σ(s_j - s_i) # NDCG = 归一化折损累积增益，是信息检索领域标准排序质量指标 # 相比Pointwise（每篇单独打分），LambdaRank直接优化排序结果

差异化排序目标 — 各产研模块独立排序策略

物流团队和广告团队关注的信号截然不同，用同一套排序权重会导致"对谁都不够好"。长期方案支持多头排序：共享底层特征提取，每个业务模块有独立的排序打分头。

产研模块优先信号类型差异化特征
零售与交易 搜推算法进展、转化率优化、个性化生成 相关性权重中 智能零售/交易平台 域加权×1.4
物流与机器人 具身智能突破、仓储自动化、路径规划 具身智能/物流域加权×1.5；时效性更重要（物流需要快速响应）
营销与增长 多模态广告技术、内容生成、用户增长 广告营销域加权×1.3；X背书来自投资人账号时额外加权
金融科技 支付监管政策、BNPL进展、风控AI 政策与专利立场权重更高；国内政策来源tier提升
前沿技术 基础模型能力演进、算力效率、安全对齐 新颖性权重提升（关注前沿突破而非成熟应用）；顶尖研究者背书更重要
🤝
与京东内部算法团队的协同机会：上述架构（双路召回、LambdaRank精排、多目标融合）与业界主流推荐系统架构一致，可直接与京东搜推算法团队对齐技术选型，避免重复建设，并借助内部基础设施加速落地。

判断力飞轮 — 系统如何越用越准

专家反馈 → 训练数据 → 精排模型迭代 → 排序结果更贴近业务 → 专家信任度提升 → 反馈质量和数量提升。这个正向循环的关键触发点是专家愿意持续反馈，而这取决于系统展示的内容是否真的有用。因此初期宁可阈值宽松（让更多内容进入专家视野），也不要过于保守（专家看了一堆无关内容后直接放弃）。

# 飞轮各环节 KPI 建议 反馈收集阶段：每周专家有效标注 ≥ 10 条（JoySpace文档点评） 初版模型上线：累积 200 条 pairwise 标注后训练 模型质量指标：NDCG@10 在验证集上优于规则基线 ≥ 5% 专家信任指标：专家打开简报后点击阅读原文比率 ≥ 30% 业务落地指标：每月由情报触发的产研跟进议题 ≥ 3 个
